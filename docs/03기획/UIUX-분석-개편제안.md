# HAIFeR / HyperClaw UI·UX 분석 및 개편 제안

> 작성일: 2026-02-27  
> 대상: 웹 클라이언트 전체 (오피스·대시보드·업무지시·결과물·도서관·설정)

---

## 1. 현재 구조 요약

### 1.1 정보 구조 (IA)

| 메뉴 (View) | 역할 | 주요 사용자 행동 |
|-------------|------|------------------|
| **오피스** | 부서/에이전트 배치, 팀장 소집 | 부서 클릭 → 채팅, 에이전트 관리 |
| **도서관** | 스킬 검색·학습·에이전트 학습 스킬 | 검색/카테고리 → 학습 모달 → 이력 확인 |
| **대시보드** | 지표·성과·부서 진행률·미션 로그 | 한눈에 보기, CTA로 업무/미션 시작 |
| **업무지시** | 지시 생성·진행 추적·빠른 지시 | 새 지시/빠른 지시 → 실행·터미널·회의록 |
| **업무 관리** | 칸반 보드, 태스크 드래그·일괄 숨김 | 컬럼 접기, 생성, 배정, 실행 |
| **결과물** | 완료 태스크 열람·코드/테스트/로그 | 필터 → 상세 → 탭(요약/코드/테스트 등) |
| **설정** | 회사·언어·CLI·OAuth | 설정 저장, 테마 토글 |

### 1.2 디자인 시스템 (현재)

- **테마**: 다크(기본)·라이트, CSS 변수(`--th-*`) 기반.
- **타이포**: Sora(제목), IBM Plex Sans KR(본문), JetBrains Mono(코드).
- **색상**: `--th-bg-primary`, `--th-text-*`, `--th-border`, `--th-focus-ring` 등 일원화. 다크는 보라·시안 계열 그라데이션, 라이트는 따뜻한 베이지 계열.
- **컴포넌트 패턴**: `game-panel`(대시보드), `dlv-*`(결과물), `header-action-btn`(헤더), 카드·모달·탭 공통.

---

## 2. 강점

- **일관된 테마**: 다크/라이트 전환과 변수 체계가 잘 잡혀 있음.
- **다국어**: ko/en/ja/zh 대응, `useI18n`·`t()` 패턴 통일.
- **접근성 일부 반영**: `aria-label`, `aria-expanded`, 포커스 링(`--th-focus-ring`), 키보드 동작(엔터/스페이스) 등.
- **접기·저장**: 대시보드 블록·도서관 가이드/에이전트 학습 스킬·업무 관리 컬럼 접기, localStorage로 유지.
- **역할 분리**: 업무지시(지시 중심) vs 업무 관리(칸반) vs 결과물(산출물)이 명확히 구분됨.

## 3. 개선이 필요한 영역

### 3.1 네비게이션·정보 구조

| 항목 | 현상 | 제안 |
|------|------|------|
| **메뉴 수** | 7개로 다소 많음. 모바일에서 전부 노출 시 부담 | 그룹화(예: "업무" 아래 업무지시·업무관리·결과물) 또는 메뉴 2단 구조 검토 |
| **헤더 액션** | 업무·의사결정·에이전트·보고서 등 버튼이 많아 뷰별로 필요 차이 무시 | 뷰별로 핵심 액션만 노출(예: 결과물 뷰에서는 "보고서" 강조) |
| **브레드크럼 부재** | 현재 뷰만 제목으로 표시, 상위 맥락 없음 | 설정·결과물 상세 등 깊은 화면에 브레드크럼 또는 "뒤로" 맥락 표시 |

### 3.2 시각적 일관성

| 항목 | 현상 | 제안 |
|------|------|------|
| **페이지 타이틀 스타일** | 오피스(header+border), 대시보드(게임 헤더), 결과물(hero), 도서관(제목+설명) 등 제각각 | 공통 "페이지 헤더" 컴포넌트: 제목+부제+선택 CTA, 스타일 가이드 정리 |
| **빈 상태(Empty State)** | `dlv-empty`, 대시보드 빈 로그, 도서관 등 문구·아이콘·CTA가 화면마다 다름 | 공통 EmptyState 컴포넌트(아이콘+제목+설명+선택 CTA), 톤 통일 |
| **로딩 상태** | 스피너·문구가 컴포넌트마다 상이 | 공통 LoadingBlock/Skeleton, 가능한 곳에 스켈레톤 적용 |
| **버튼 계층** | primary/secondary/ghost가 파일마다 인라인 스타일로 정의 | `.btn-primary`, `.btn-secondary`, `.btn-ghost` 등 유틸리티 또는 Button 컴포넌트로 통일 |

### 3.3 반응형·모바일

| 항목 | 현상 | 제안 |
|------|------|------|
| **사이드바** | lg에서만 상시 표시, 모바일은 오버레이. 접힌 폭(w-16) 고정 | 태블릿(md)에서도 접기/펼치기 선택 가능하도록 브레이크포인트 검토 |
| **마스터-디테일** | 업무지시·결과물에서 목록/상세 전환 시 목록이 완전히 숨겨짐 | 필요 시 슬라이드/드로어 형태로 "목록 미리보기" 유지 또는 스와이프 제스처 |
| **헤더** | 모바일에서 아이콘만 보이는 버튼 많음, 툴팁 없으면 의미 파악 어려움 | `title`/`aria-label` 필수 점검, 필요 시 하단 툴바 또는 "더보기" 메뉴로 그룹화 |
| **테이블·칸반** | 업무 관리 칸반이 가로 스크롤 위주 | 컬럼 접기 기본값·우선순위 제안, 작은 화면에서 리스트 뷰 옵션 검토 |

### 3.4 접근성(A11y)

| 항목 | 현상 | 제안 |
|------|------|------|
| **포커스 순서** | 모달·드로어 열릴 때 포커스 트랩/복귀가 일부만 적용될 가능성 | 모달/슬라이드 패널 open 시 포커스 트랩, 닫기 시 이전 포커스 복귀 공통화 |
| **스크린 리더** | 동적 영역(실시간 로그·알림)에 live region 미적용 가능성 | `aria-live="polite"` 영역 정의, 중요 알림만 읽히도록 |
| **대비** | `--th-text-muted` 등이 배경 대비 부족할 수 있음 | WCAG 2.1 AA 대비 검사(4.5:1 본문, 3:1 큰 텍스트), 변수 조정 |
| **폼 에러** | 에러 메시지와 필드 연결(aria-describedby) 미비 가능성 | 필드별 `id`·`aria-describedby`·`aria-invalid` 일관 적용 |

### 3.5 성능·체감

| 항목 | 현상 | 제안 |
|------|------|------|
| **초기 로딩** | 로딩 시 "하이퍼클에이전트 로딩 중..." 한 화면만 있음 | 스플래시/브랜드 일관, 필요 시 단계별 진행 표시 |
| **긴 목록** | 결과물·스킬·태스크 목록이 한 번에 렌더링될 수 있음 | 가상 스크롤(react-window 등) 또는 페이지네이션 도입 검토 |
| **탭 전환** | 결과물 상세에서 탭 전환 시 매번 컴포넌트 마운트 | 이미 본 탭은 유지하되 보이기만 전환(display/visibility) 또는 lazy+캐시 |

### 3.6 콘텐츠·카피

| 항목 | 현상 | 제안 |
|------|------|------|
| **뷰 제목 불일치** | useAppLabels의 skills가 "문서고"로 되어 있음. 사이드바는 "도서관" | 라벨 키 일원화(도서관/Library), 모든 뷰 제목이 사이드바·헤더와 동일한 문자열 사용 |
| **에러·빈 상태 문구** | 기술적 표현("No execution logs")이 사용자 친화적이지 않을 수 있음 | 사용자 관점 문구로 정리(예: "아직 실행 기록이 없어요") |
| **가이드** | 도서관 "사용법 및 가이드"만 접기 가능. 다른 뷰에는 첫 방문용 힌트 없음 | 업무지시·결과물·대시보드에 짧은 "이 화면은 이렇게 쓰세요" 접기 블록 검토 |

## 4. 개편 우선순위 제안

### Phase 1: 빠른 정리 (1–2주)

1. **페이지 헤더 통일**  
   결과물·업무지시·도서관 등에 공통 `PageHeader`(제목+부제+선택 CTA) 적용, 스타일 변수만 다르게.
2. **라벨 정리**  
   "문서고" → "도서관" 등 뷰 제목·사이드바·헤더 일치, 필요 시 ko/en 문구 사용자 친화적으로 수정.
3. **빈 상태·로딩 공통화**  
   `EmptyState`, `LoadingBlock`(또는 스켈레톤) 컴포넌트 도입 후 기존 인라인 빈 상태/로딩을 점진 교체.
4. **접근성 점검**  
   모달/패널 포커스 트랩·복귀, 버튼/링크 `aria-label`·`title` 보완.

### Phase 2: 구조·경험 (2–4주)

1. **네비게이션 그룹화**  
   사이드바에서 "업무" 그룹(업무지시·업무 관리·결과물) 묶음 표시 또는 2단 메뉴.
2. **헤더 액션 컨텍스트화**  
   뷰별로 표시할 액션 정의(예: 결과물에서는 "보고서" 강조, 업무지시에서는 "업무" 강조).
3. **반응형 보강**  
   마스터-디테일 레이아웃(업무지시·결과물)의 태블릿/모바일 동작 정리(드로어·스와이프 등).
4. **버튼·카드 스타일 시스템**  
   `.btn-*`, 카드 variant(기본/강조/위험)를 스타일 가이드와 컴포넌트로 정리.

### Phase 3: 심화 (1–2개월)

1. **성능**  
   긴 목록 가상화/페이지네이션, 결과물 탭 전환 시 렌더 비용 절감.
2. **온보딩·가이드**  
   첫 방문 또는 뷰별 짧은 도움말(접기 가능), 필요 시 단계별 튜토리얼 검토.
3. **WCAG AA 목표**  
   색상 대비·포커스·키보드·스크린 리더 요구사항 정리 후 단계별 조치.

## 5. 전체 레이아웃 개편 제안

### 5.1 현재 레이아웃 구조 (코드 기준)

| 영역 | 현재 구현 | 비고 |
|------|-----------|------|
| **App Shell** | `flex h-[100dvh]` · 좌측 사이드바 + `main`(flex-1, overflow-y-auto) | 전역 구조는 단순·명확 |
| **사이드바** | `lg` 이상에서만 상시 노출, `w-48`(펼침)/`w-16`(접힘). 모바일은 `fixed` 오버레이 + 딤드 | 7개 메뉴 플랫, 그룹 없음 |
| **헤더** | `sticky` 상단, 뷰 제목 + 업무/의사결정/에이전트/보고서/공지/테마 등 동일 버튼 세트 | 뷰별 차등 없음 |
| **메인 콘텐츠** | `p-3 sm:p-4 lg:p-6` 내부에 뷰별 컴포넌트 전환 | 패딩만 공통, 페이지 헤더 스타일 불일치 |
| **업무지시/결과물** | 좌(목록) + 우(상세/폼). `md` 미만에서는 선택 시 목록 `hidden`, 상세만 전체 너비 | 마스터-디테일에서 목록 맥락 상실 |

### 5.2 레이아웃 개편 방향

#### A. App Shell (유지 + 보강)

- **유지**: `app-shell` = 사이드바 + 메인 2열 구조, `100dvh` 고정 높이.
- **보강**:
  - 메인 영역에 `min-w-0` 유지(플렉스 오버플로 방지).
  - 필요 시 메인 내부를 **상단 헤더/배너 | 스크롤 콘텐츠**로만 나누어, 뷰별로 "페이지 헤더 + 본문"이 일관되게 들어가도록 슬롯화 검토.

#### B. 사이드바 — 그룹화 및 2단 구조

- **목표**: 메뉴 7개 부담 완화, "업무" 흐름(지시 → 관리 → 결과물) 인지 용이.
- **안 1 — 그룹 라벨만 추가**  
  - 상단: 오피스 · 도서관 · 대시보드 (진입/개요).  
  - 구분선 + "업무" 그룹: 업무지시 · 업무 관리 · 결과물.  
  - 하단: 설정.  
  - 접힌 상태(w-16)에서는 아이콘만, 그룹 구분은 시각적(간격/구분선)으로만.
- **안 2 — 2단 메뉴(펼침 시)**  
  - 1차: "업무" 하나의 항목(아이콘+라벨).  
  - 클릭 시 2차: 업무지시 / 업무 관리 / 결과물 서브메뉴 표시(인디케이트 또는 슬라이드).  
  - 데스크톱에서 서브메뉴는 사이드바 확장 또는 옆 패널로 표시.
- **브레이크포인트**: 현재 `lg`(1024px)에서만 상시 사이드바. **태블릿(md, 768px)** 에서도 "사이드바 접기/펼치기" 선택 가능하도록 옵션 검토(저장은 localStorage).

#### C. 헤더 — 뷰별 액션 + 브레드크럼

- **뷰별 액션**:  
  - 공통 유지: 테마, (모바일) 햄버거·더보기.  
  - 뷰별로 표시할 액션만 노출: 예) 결과물 → "보고서" 강조, 업무지시/업무관리 → "업무" 강조, 오피스 → "에이전트/공지" 강조.  
  - 정의 테이블 예: `{ view: 'deliverables', primary: ['report'], secondary: ['tasks','announcement'] }`.
- **브레드크럼**:  
  - 설정(하위 화면), 결과물 상세, 업무지시 상세 등 **깊이 2단계 이상**인 화면에만 적용.  
  - 예: `결과물 > [태스크 제목]`, `설정 > (현재 섹션)`.  
  - 구현: 헤더 왼쪽 또는 제목 위에 `PageHeader`와 연동된 작은 브레드크럼 컴포넌트.

#### D. 메인 콘텐츠 영역

- **공통 패딩**: 현재 `p-3 sm:p-4 lg:p-6` 유지.
- **페이지 헤더 일원화**: 각 뷰 상단을 공통 `PageHeader`(제목 + 부제 + 선택 CTA)로 교체해, 결과물 hero / 도서관 / 업무지시 등 스타일을 변수만으로 구분.
- **최대 너비(선택)**: 대시보드·도서관 등 넓은 카드 그리드에 `max-w-7xl mx-auto` 적용해 초대형 화면에서 가독성 확보.

#### E. 마스터-디테일 (업무지시 · 결과물)

- **현상**: `md` 미만에서 상세 열면 목록이 완전히 `hidden`, "목록으로" 버튼으로만 복귀.
- **개선 1 — 드로어/슬라이드**:  
  - 상세를 오른쪽에서 슬라이드인 패널(드로어)로 표시.  
  - 목록은 뒤에 반투명 딤드로 남겨 두고, 상세 패널 닫기 시 목록이 다시 보이게 함.
- **개선 2 — 목록 미리보기 유지**:  
  - 상세 열릴 때 목록을 좁은 스트립(예: 48px~64px) 또는 아이콘+제목 한 줄로 왼쪽에 유지.  
  - 탭/스와이프로 "목록 확장 ↔ 상세" 전환.
- **접근성**: 드로어/패널 열릴 때 포커스 트랩, 닫기 시 이전 포커스 복귀(§3.4와 연동).

### 5.3 반응형 브레이크포인트 정리

| 구간 | 사이드바 | 헤더 액션 | 마스터-디테일 |
|------|----------|-----------|----------------|
| **&lt; md (768px)** | 햄버거 → 오버레이 | 아이콘+더보기 메뉴, aria-label 필수 | 목록 ↔ 상세 전환 또는 드로어 |
| **md ~ lg (768–1023)** | (현재) 오버레이 동일 또는 접기 옵션 실험 | 텍스트 버튼 일부 노출 | 목록 + 상세 2열 가능 |
| **lg+ (1024px)** | 상시 표시, 접기 시 w-16 | 전체 액션 노출 | 2열 유지 |

### 5.4 레이아웃 개편 적용 순서

1. **Phase 1**  
   - 페이지 헤더 통일(`PageHeader`) → 메인 콘텐츠 구조가 각 뷰에서 일관됨.  
   - 헤더 액션은 우선 `aria-label`/`title` 보강만.
2. **Phase 2**  
   - 사이드바 그룹화(안 1 또는 안 2).  
   - 헤더 뷰별 액션 테이블 도입 및 조건부 렌더링.  
   - 깊은 화면에 브레드크럼 추가.  
   - 마스터-디테일: 모바일에서 드로어 또는 목록 미리보기 중 하나 적용.
3. **Phase 3**  
   - 태블릿(md) 사이드바 접기/펼치기 옵션.  
   - 대형 화면 max-width·그리드 정리.

## 6. 전체 스타일 개편 제안

### 6.1 현재 스타일 요약

| 영역 | 현재 구현 | 비고 |
|------|-----------|------|
| **테마 변수** | `--th-*` (bg, text, border, focus-ring, card, panel, input, scrollbar 등) | 다크/라이트 일원화 잘 됨 |
| **타이포** | Sora(제목), IBM Plex Sans KR(본문), JetBrains Mono(코드) | 유지 적합 |
| **배경** | body·app-shell 그라데이션(다크: 블루/시안, 라이트: 블루/앰버) | 분위기 명확 |
| **컴포넌트 클래스** | `header-action-btn`, `sidebar-nav-item`, `game-panel`, `dash-card`, `postit-card`, `taskboard-*`, `theme-toggle-btn` 등 | 뷰/기능별로 클래스 산재 |
| **버튼** | `header-action-btn-primary/secondary` + 인라인/Tailwind 혼용 | primary/secondary/ghost/danger 계층 미통일 |
| **라이트 모드** | `[data-theme="light"]` 하위 다수 선택자로 slate/gray/zinc/색상 오버라이드 | 수백 줄 규모, 유지보수 부담 |
| **애니메이션** | 대시보드(kpi-pop, bubble-float, crown-wiggle 등), 스킬(learn-*, unlearn-*), agent-bounce 등 | `prefers-reduced-motion` 대응 있음 |

### 6.2 스타일 개편 방향

#### A. 디자인 토큰·변수 보강

- **시맨틱 확장**:  
  - 버튼용: `--th-btn-primary-bg`, `--th-btn-primary-text`, `--th-btn-secondary-border` 등(선택).  
  - 또는 기존 `--th-*`만 쓰고, 버튼/카드는 **유틸리티 클래스**로 조합.
- **대비(WCAG)**  
  - `--th-text-muted` 대비 검사(4.5:1 본문, 3:1 큰 텍스트).  
  - 부족 시 라이트/다크 모두에서 값 조정.
- **강조색 일원화**  
  - 성공/경고/위험/정보에 `--th-success`, `--th-warning`, `--th-danger`, `--th-info` 도입 시 라이트 오버라이드 단순화에 도움.

#### B. 버튼·카드 스타일 시스템

- **버튼**  
  - `.btn-primary`, `.btn-secondary`, `.btn-ghost`, `.btn-danger` (및 size: sm/md/lg)를 `index.css` 또는 공통 Button 컴포넌트로 정의.  
  - `header-action-btn-*`는 이 계층을 사용하거나, 헤더 전용 variant로 매핑.  
  - 인라인 `style={{ }}` 및 Tailwind 색상 클래스(예: `bg-blue-600`)를 점진적으로 이 계층으로 교체.
- **카드**  
  - 기본: `--th-card-bg`, `--th-card-border` 사용(현재 `dash-card`, `taskboard-column` 등과 유사).  
  - variant: 기본 / 강조(accent border) / 위험(삭제 등) — 클래스 하나로 구분.  
  - 결과물 `dlv-panel`, 업무지시 패널 등이 동일 카드 베이스 + variant로 통일.

#### C. 페이지 헤더·빈 상태 스타일 통일

- **PageHeader**  
  - 결과물 `deliverables-hero`, 도서관 상단, 업무지시 상단 등이 **동일한 구조**(제목+부제+CTA)와 **공통 클래스**(예: `.page-header`, `.page-header__title`, `.page-header__desc`)를 사용.  
  - 배경/그라데이션은 선택(뷰별로 테마 변수만 다르게 적용 가능).
- **EmptyState**  
  - 아이콘+제목+설명+CTA 레이아웃과 간격·타이포를 한 세트 클래스로 정의.  
  - `dlv-empty`, 대시보드 빈 로그, 도서관 빈 목록 등이 동일 클래스 사용.

#### D. 애니메이션·모션 정책

- **유지**: `prefers-reduced-motion: reduce` 시 애니메이션/트랜지션 최소화(현재 적용됨).
- **정리**:  
  - 장식용 애니메이션(예: crown-wiggle, bubble-float)은 유지하되, 과한 요소는 옵션 또는 라이트 모드에서만 완화 검토.  
  - 피드백용(로딩 스피너, 버튼 클릭, 탭 전환)은 짧고 일관된 duration(예: 150ms~250ms) 권장.
- **신규**: 로딩/스켈레톤용 공통 키프레임(예: `shimmer`)은 이미 있음 — `LoadingBlock`/스켈레톤 컴포넌트에서 재사용.

#### E. 라이트 모드 오버라이드 정리(선택)

- **현상**: Tailwind 색상 클래스(slate/gray/blue/amber 등)를 라이트에서 전역 오버라이드하여 수백 줄 유지.
- **방향**  
  - 신규/수정 구간은 가능한 한 `var(--th-*)`와 공통 클래스만 사용해, 라이트 오버라이드 추가를 줄임.  
  - 기존 오버라이드는 당장 제거하지 않고, 점진적으로 `--th-*` 기반으로 이전 시 해당 선택자 제거.  
  - 컴포넌트 단위로 “이 컴포넌트는 테마 변수만 사용” 규칙을 적용하면 정리 범위가 명확해짐.

#### F. 그라데이션·글래스·네온 톤

- **현재**: 다크에서 시안/블루/보라 그라데이션, 대시보드 타이틀/랭킹 그라데이션, 게임 패널 글래스 효과.
- **제안**  
  - 톤은 유지하되, 그라데이션 사용처를 **헤더·대시보드·강조 영역**으로 한정해 일관성 유지.  
  - 일반 목록/폼/테이블은 평면 색(`--th-bg-surface`, `--th-card-bg`) 위주로 해서 가독성 우선.  
  - 라이트 모드에서는 그라데이션을 더 약하게(또는 단색에 가깝게) 적용해 과한 장식 방지.

### 6.3 스타일 개편 적용 순서

1. **Phase 1**  
   - 버튼 유틸리티 `.btn-primary` / `.btn-secondary` / `.btn-ghost` (및 size) 정의.  
   - PageHeader·EmptyState 공통 구조+클래스 정의 후, 결과물·도서관 등 1~2개 뷰에 적용.  
   - `--th-text-muted` 등 대비 점검 및 필요 시 변수 값만 조정.
2. **Phase 2**  
   - 카드 variant(기본/강조/위험) 정리, 결과물/업무지시 패널에 적용.  
   - 헤더 액션 버튼을 공통 버튼 계층과 연동(또는 헤더 전용 variant).  
   - 신규/수정 컴포넌트는 `--th-*` + 공통 클래스만 사용하도록 가이드.
3. **Phase 3**  
   - 라이트 모드 오버라이드 점진 축소(컴포넌트 단위 이전).  
   - 강조색 시맨틱 변수(`--th-success` 등) 도입 검토.  
   - 그라데이션/글래스 사용처 문서화 및 가이드라인 정리.
